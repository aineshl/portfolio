The Problem

Controller configurations were built in a brittle Excel macro workbook. Large jobs took nearly two hours of senior engineering time. Hardware constraints lived in engineers' heads, not in the tool — and every error became a support ticket. The process didn't scale, and it was costing the team time they didn't have.

Phase 1 — PM

I scoped the product, defined the ERP data model, and managed an external developer to build the first version — a validated web configurator integrated with the company's ERP system (Microsoft Business Central). Key architecture decision: keep configuration metadata out of the ERP core to avoid bloat, while keeping BC as the source of truth for products, pricing, and fulfillment.

I used MagicPatterns for rapid UX prototyping before development, aligning stakeholders on the interface before a single line of code was written. Excel was deprecated on launch day — no rollback path.

Phase 2 — PM + Developer + QA

After launch, I took over the codebase entirely and rebuilt the UI for three distinct user personas — each with different mental models and workflows:

  • Production team — speed and zero ambiguity. Configures units at volume. Every extra click is friction that compounds across hundreds of units.
  • Application engineers — full technical control. Designs complex multi-zone systems with multiple controllers, channels, and priorities.
  • Customers — clarity without product knowledge. Reviews and approves configurations without needing to know what an FCS controller is.

I also built and maintained the Supabase backend (auth, data persistence, project state) and the full Business Central integration.

Testing & Hardware Validation

The configurator's output is a .cfg file that programs a physical controller in the field. A misconfigured file means a misconfigured installation. I treated quality accordingly:

  • Wrote 20 structured test cases covering edge cases, constraint violations, and standard production configurations
  • Used AI-assisted regression testing to validate .cfg output after every UI change or bug fix
  • Built a bug ticketing system — personally investigated and resolved every issue filed
  • Personally loaded exported .cfg files onto physical controllers and validated behavior for every test case

Every change I shipped was validated against the physical controller a technician would install in a parking garage or a distillery. That's the bar I held myself to.

Results

  • 75%+ reduction in configuration time — large jobs from ~2 hours to under 30 minutes
  • Manual configuration eliminated across the production team
  • Full internal adoption: production, application engineers, and customer-facing workflows
  • Excel tool fully deprecated with no rollback
  • Platform now expanding to cover detector configuration — scope continues to grow
