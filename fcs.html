The problem

Controller configurations at CET were built inside a brittle Excel macro workbook that broke regularly under real-world complexity. Large configurations — the kind needed for multi-zone parking garages or complex industrial facilities — took nearly two hours of manual engineering time. Hardware constraints on what an FCS controller could actually support lived in people's heads, not in the tool.

Every error was a support ticket. Every manual configuration was senior engineering time that shouldn't have been needed. This directly conflicted with CET's core promise of fast service and support.

Phase 1 — PM

I scoped and managed an external developer to build a validated web configurator integrated with Microsoft Business Central — CET's quote-to-cash ERP.

The critical architecture decision was the data model: I kept config-specific metadata out of the ERP core to avoid bloat, while keeping Business Central as the single source of truth for products, pricing, and fulfillment. The same model supports system design, BOMs, quoting, and configuration simultaneously. Excel was fully deprecated on launch day — no rollback option.

I also designed the UX and user flows from scratch, partnered with engineering to validate .cfg output against real controllers, and owned the product end-to-end through delivery.

Phase 2 — PM + Developer

After launch I took over the Lovable codebase entirely and rebuilt the UI from scratch for three distinct user personas, each with fundamentally different needs:

  • Production team — needs speed and zero ambiguity. Configures units at volume. Every extra click is friction.
  • Application engineers — needs full technical control. Designs complex multi-zone systems with many transmitters, channels, and priorities.
  • Customers — needs clarity without product knowledge. Reviews and approves configurations without knowing what an FCS-128 is.

I personally built and maintained the Supabase backend (auth, data persistence, project state) and the Business Central integration.

Testing and validation

The configurator's critical output is a .cfg file — the binary configuration that programs a physical FCS controller. A wrong file means a misconfigured controller in the field. I treated this with the same rigor as hardware validation:

  • Wrote 20 structured test cases covering edge cases, constraint violations, and standard production configurations
  • Used AI-assisted validation inside Lovable to run regression checks after every UI change or bug fix — ensuring the .cfg export stayed functionally identical
  • Set up a bug ticketing system for internal QA — personally investigated and resolved every bug filed
  • Personally loaded exported .cfg files onto physical FCS controllers and validated field behavior for every test case

The hardware validation loop was the hardest part to get right and the most important. Every change I shipped — a UI tweak, a Business Central sync update, a bug fix — was ultimately validated against the physical controller that a technician would install in a parking garage or distillery. That's the standard I held the work to.
